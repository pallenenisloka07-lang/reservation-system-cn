miniproject for Computer Networks course
# Secure Concurrent Reservation System
A network-based reservation system built with Python sockets + TLS that supports concurrent clients, atomic booking, double-booking prevention, crash recovery, and stress testing.
## Project Files
- `server.py`
  - Runs the TLS-enabled reservation server.
  - Maintains shared seat state in memory.
  - Enforces atomic updates using a mutex lock.
  - Implements request protocol parsing and response generation.
  - Persists state via journal + snapshot for recovery after restart/crash.
- `server.crt`, `server.key`
  - TLS certificate and private key used by the server.
  - Auto-generated by `server.py` if missing.
- `booking_journal.jsonl`
  - Write-ahead log of mutating operations (`BOOK`, `CANCEL`, `CLEAR`).
  - Used for replay/recovery.
- `state_snapshot.json`
  - Snapshot of latest seat map + version.
  - Loaded on startup for faster recovery.
- `.gitignore`
  - Excludes generated runtime artifacts (`.pyc`, cert/key, journal, snapshot).
---
## Seat Model
- Total seats: `SEAT_1` to `SEAT_5`
- A seat value is either:
  - booked user id (example: `USER_ALICE`), or
  - `None` (available)
---
## Concurrency and Atomicity
`server.py` uses:
- `state_lock = threading.Lock()`
- All request handling that reads/writes booking state is done inside `with state_lock:`.
This guarantees:
- only one thread can modify seat state at a time,
- no race conditions during booking,
- prevention of double booking under concurrent clients.
---
## Protocol (Request / Response)
All messages are line-based (`\n` terminated), fields separated by `|`.
### Requests
- `BOOK|REQ_ID|SEAT_ID|USER_ID|CLIENT_TS`
- `CANCEL|REQ_ID|SEAT_ID|USER_ID|CLIENT_TS`
- `QUERY|REQ_ID|SEAT_ID|USER_ID|CLIENT_TS`
- `LIST|REQ_ID|CLIENT_TS`
- `CLEAR|REQ_ID|USER_ID|CLIENT_TS`
### Responses
- `RESP|REQ_ID|STATUS|SERVER_TS|CODE|MESSAGE|PAYLOAD`
Examples of `STATUS`:
- `OK`
- `FAIL`
- `ERROR`
Examples of `CODE`:
- `BOOKED`, `ALREADY_BOOKED`
- `CANCELLED`, `NOT_BOOKED`, `NOT_OWNER`
- `LIST_OK`, `QUERY_OK`, `CLEARED`
- `BAD_FORMAT`, `INVALID_SEAT`, `UNKNOWN_ACTION`
---
## Idempotency (Duplicate Retry Safety)
Server keeps an in-memory `request_cache` keyed by `REQ_ID`.
If same request id is received again, server returns cached response instead of re-applying mutation.
---
## Failure Handling and Recovery
### Write path for state-changing operations
For `BOOK`, `CANCEL`, `CLEAR`:
1. Validate request under lock.
2. Append entry to `booking_journal.jsonl` and fsync.
3. Apply in memory.
4. Save `state_snapshot.json` atomically (`.tmp` + replace) and fsync.
### Startup recovery
On server start:
1. Load snapshot (if present).
2. Replay journal entries newer than snapshot version.
3. Save fresh snapshot.
This makes state durable across restarts/crashes.
---
## Where SSL/TLS Is Used
### In `server.py`
- `context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)`
- `context.load_cert_chain(certfile="server.crt", keyfile="server.key")`
- `secure_server = context.wrap_socket(server_socket, server_side=True)`
The server accepts TLS connections only.
> Note: This is acceptable for local development only. In production, enable cert verification and hostname checks.

## Run Instructions
## 1) Start server
```bash
python3 server.py
```
To bind explicitly (useful on cloud/remote hosts):
```bash
python3 server.py --host 0.0.0.0 --port 65432
```
---
## Running Server Anywhere (Not Just Localhost)
1. Run `server.py` on the target machine (cloud VM, another laptop, etc.) with `--host 0.0.0.0`.
2. Open inbound TCP access to the chosen port (default `65432`) in firewall/security-group rules.
---
## Typical Demo Flow
1. Start server.
2. Book one seat with User A.
3. Try booking same seat with User B (should fail as already booked).
4. Run `LIST` to inspect seat table.
5. Run `CLEAR` to reset all seats.
6. Run `LIST` again (all seats available).
7. Run `stress_test.py` to validate concurrent behavior.
---
## Notes
- Seat capacity is currently fixed at 5 (`SEAT_COUNT = 5`).
- `CLEAR` is a global reset operation intended for testing/admin use.
- Runtime files (`server.crt`, `server.key`, journal/snapshot) are auto-generated and ignored by git.
